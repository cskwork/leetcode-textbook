# Changelog - 2026-07-05

## Decision: build a pattern-structured Java LeetCode textbook

**Context.** The user requested a beginner-friendly "textbook" of 100 LeetCode problems (Easy ->
Medium) in a new folder, with detailed teaching notes, pattern recognition, and pseudocode
shown alongside every Java solution.

**Options considered.**
1. Language: Python vs JavaScript vs Java vs C++ vs Python+second. **Chose Java** per user.
2. Folder location: inside PARA/Resource vs a separate Projects folder. **Chose
   `/Users/danny/Documents/PARA/Resource/leetcode-textbook/`** per user.
3. Curation: pattern-structured vs Blind 75+25 vs NeetCode 150 subset vs top-100-by-acceptance.
   **Chose pattern-structured** (NeetCode-style 16-pattern map) per user.

## Architecture decisions

### Pattern count and distribution
- **16 patterns**, totalling exactly 100 problems. Distribution: 3-9 problems per pattern.
- Three officially-Hard problems were included as section capstones because they are the
  canonical teaching instances of their pattern: LC 76 (Min Window Substring, Sliding Window),
  LC 212 (Word Search II, Tries), and LC 5 (Longest Palindromic Substring, 1-D DP -- LC 5 is
  officially Medium so this is only a note).
- Two problems deliberately appear in two patterns to demonstrate pattern overlap: LC 347
  (bucket-sort in Pattern 1, heap in Pattern 9) and LC 11 (two-pointer in Pattern 2, greedy in
  Pattern 12). Each lives in a distinct folder (`0347-top-k-frequent-heap/`,
  `0011-container-greedy/`).

### Per-problem layout
- Fixed three-file structure: `README.md` (teaching notes), `Solution.java` (clean LeetCode
  solution), `SolutionTest.java` (runnable `main`-method test, no JUnit).
- README follows a strict section order: Problem -> Intuition -> Pseudocode -> Java ->
  Complexity -> Dry-Run -> Common mistakes -> Related problems.
- The **pseudocode precedes Java** rule is the pedagogical core and was made non-negotiable in
  every subagent contract. The pseudocode is language-neutral (no `HashMap`, no `int[]`, no
  `for(int i=0;...)`).
- Node classes (`ListNode`, `TreeNode`, `TrieNode`) are top-level in `Solution.java` and shared
  with the test via joint compilation -- no redefinition, no separate file.

### Test framework choice
- Plain `main`-method tests, no JUnit. Rationale: zero-install verification -- a beginner can
  `javac Solution.java SolutionTest.java && java SolutionTest` with only the JDK installed.
- 679 total test cases across the 100 problems. Each problem has 3-6 cases covering LeetCode's
  public examples plus at least one edge case.

## Orchestration decisions

- **Subagent-driven, one subagent per pattern.** 16 patterns were generated by 16 independent
  `general` subagents in 4 parallel waves of 4. Each subagent received the same contract
  (`TEMPLATE.md`), its problem list, pattern-specific must-haves (e.g. LC 238 without division,
  LC 704 with overflow-safe mid), and a verification requirement (compile + run + report
  output).
- **Wave structure** (4+4+4+4): foundational patterns first, DP last. Waves ran in parallel
  internally; waves were sequential so quality issues in early patterns could be caught before
  committing to later ones. In practice no wave produced a systemic defect.
- **Ground-truth verification after every wave and at the end.** A final shell pass compiled and
  ran all 100 problem test suites: **100/100 pass**. No test was weakened; several subagents
  reported correcting their own test expectations when the solution was right and the expected
  literal was wrong (the right discipline).

## Alternatives rejected

- **Single mega-generation in one prompt.** Rejected: too large for a single coherent output;
  per-pattern subagents gave each section a dedicated context and let problems compile in
  parallel.
- **Bucket-sort OR heap for LC 347 (choose one).** Rejected in favor of teaching both -- the
  hashing pattern and the heap pattern both have something to say about top-K, so the problem
  appears in both sections with different approaches.
- **Expand to Hard problems.** Rejected by design. The book's stated scope is Easy/Medium.
  Hard problems are mostly Medium patterns with extra constraints; the 100 Easy/Medium problems
  here cover the necessary patterns.
